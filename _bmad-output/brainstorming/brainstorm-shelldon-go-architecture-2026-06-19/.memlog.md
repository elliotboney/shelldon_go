- (technique) started First Principles — strip the v2 spine to invariants, challenge each must from Go's vantage
- (idea by coach) The fork-server (v2's 'non-retrofittable load-bearing bet') protected bounded RAM — Go gives that free, so the bet's VALUE is now zero; the question it answered re-emerges as a FAULT question, not a memory one
- (idea by coach) Reframe AD-9's '≤1 worker': in Go RAM isn't the constraint, so keep ≤1 not for megabytes but for COHERENCE (one train of thought, no two turns racing the memory writer) + COST (budget). The limit is about the pet having one mind, not bytes
- (insight by coach) The v2 architecture optimized for PYTHON's failure modes (OOM, GIL, cheap fork). The Go architecture should optimize for GO's failure modes: goroutine leaks, shared-memory races, panic propagation. Different enemies → different shape
- (idea by coach) Third isolation primitive: the worker as a WASM sandbox via wazero (pure-Go WASM runtime). Prompt-assembly+tool-selection runs in WASM linear memory with NO ambient authority — capabilities passed in. Sits between goroutine (no isolation) and subprocess (OS isolation), stays single-binary
- (insight by coach) Vault requirement restated precisely: not 'separate process' but 'the thing touching web-influenced text cannot read secrets'. Goroutine = insufficient BY DEFINITION (shared address space). So the real requirement is ADDRESS-SPACE separation; the only question is which kind (process vs WASM linear memory)
- (idea by coach) Inversion: don't isolate the worker — isolate the SECRETS. A tiny 'secrets-keeper' process holds vault; broker queries it; everything else is one Go process. Blast radius of a single-process collapse becomes 'everything EXCEPT secrets' — a smaller, sharper boundary than 'worker vs world'
- (idea by coach) Two of the THREE multi-process justifications evaporate in Go: (1) bounded RAM = GC free, (2) reflex-responsiveness-during-a-turn = reflexes are just ticker goroutines, alive for free. ONLY security remains. The spine was 3-legged; Go kicks out 2 legs
- (idea by coach) Provocative: does a single-user desk pet even need an OS-uid vault wall? Attack = injected text → worker calls a tool that leaks vault. If worker can't read vault AND can't make arbitrary egress (broker mediates), the attack needs TWO failures. The uid wall may be belt-and-suspenders on top of capability-mediation
- (idea by coach) Secrets-never-in-worker principle: broker injects creds at the egress moment; the worker NEVER receives them. In a multi-process design this + uid wall is airtight. In single-process it's necessary-but-insufficient (shared memory) — which is itself the cleanest argument for keeping exactly ONE process boundary
- (technique) started Analogical Thinking — who else runs untrusted compute behind a trust boundary, what did they pick
- (idea by coach) Web browser analogy: a tab (untrusted JS) is a separate OS process; the browser kernel holds cookies/creds. Chrome chose PROCESS isolation precisely because shared-memory wasn't safe enough. shelldon's worker = a browser tab; core+broker = the browser kernel. This analogy votes for Option C/A
- (idea by coach) AWS Lambda analogy: untrusted user code runs in Firecracker microVMs, one per invocation, frozen base + fork-like clone. The worker-per-turn IS a Lambda invocation. But Lambda's isolation is about MULTI-TENANCY; shelldon is single-tenant — weakens the case for heavy per-turn isolation
- (idea by coach) Game-server analogy: untrusted mods run in a Lua sandbox (sandboxed interpreter), NOT a separate process — capability-restricted VM inside the host process. This is the WASM-worker idea's pedigree: embed a restricted VM, don't spawn a process. Votes for the wazero option
- (idea by coach) Payment-terminal analogy: the secure element (holds keys) is physically separate silicon from the app CPU; the app NEVER sees the key, only asks the SE to sign. That's the broker-as-secrets-keeper pattern in hardware. On a Pi, the analog is a separate process or even the Pi's OTP/TPM — broker signs/calls, worker asks
- (idea by coach) Unix daemon analogy (postfix/qmail): split a monolith into many tiny single-purpose processes each under its own uid, talking over pipes — privilege separation by Bernstein. This is literally Option A, and it's a 25-year-proven pattern for security-critical mail. shelldon's spine is qmail-shaped
- (insight by coach) The analogies SPLIT by tenancy: multi-tenant systems (browser, Lambda) spend heavily on process/VM isolation; single-tenant embedded systems (game mods, your desk pet) often accept in-process sandboxing. shelldon is single-tenant → the WASM/in-process sandbox is more defensible than the browser analogy first suggests
- (idea by coach) OpenSSH analogy: the pre-auth network-facing code runs in a chroot+unpriv 'privsep' child; the monitor process holds privileges and validates requests. Maps EXACTLY onto broker(monitor) + worker(privsep child). The validated-request channel = your Result envelope with proposed memory-ops. Steal this: worker proposes, monitor disposes
- (idea by coach) Antivirus/sandbox-detonation analogy: suspicious files detonated in a throwaway VM that's destroyed after. The 'die after one turn' instinct is the detonation pattern — but detonation is about not trusting the ARTIFACT to persist. shelldon's worker output (Result) IS trusted-after-validation, so full detonation is overkill; ephemerality buys fault-reset, not security
- (idea by coach) Capability-OS analogy (seL4/Fuchsia): no ambient authority — a component can only do what it was handed a capability for. Recast shelldon: the worker is handed a 'call-the-model' capability and a 'read-history' capability, nothing else. No vault capability = can't read vault, even same-process IF the language enforced capabilities. Go doesn't, but wazero host-functions DO
- (technique) started Constraint Shifting — warp the givens (RAM, network, hardware, time) and see what architecture falls out
- (idea by coach) Shift RAM to 4GB (Pi 4): the entire memory-frugality apparatus is pointless; you'd run one fat process and never think about it. This proves the frugality machinery is HARDWARE-CONTINGENT, not essential — design so it's a config/build flag, not baked into the architecture's bones
- (idea by coach) Shift RAM DOWN to 256MB: now even Go's ~5-10MB runtime floor PER PROCESS matters, and Option A's 5 processes = 5 runtimes = 50MB of pure overhead. Tight RAM ARGUES FOR single-process collapse (one runtime). The frugality goal and the isolation goal PULL OPPOSITE — name that tension explicitly
- (insight by coach) Key tension surfaced: more processes = more security isolation BUT more Go-runtime overhead (each process pays the 5-10MB floor). On 512MB this caps how many processes you can afford. Option C (2 processes) is the sweet spot precisely because it's ONE security boundary at the cost of ONE extra runtime floor
- (idea by coach) Shift network to ZERO (offline week): broker's provider chain must degrade to Ollama-LAN or reflex-only. This means the broker needs a 'no brain available' state that's FIRST-CLASS, not an error path. Design the pet to be charming when brainless — offline is a personality mode, not a failure
- (idea by coach) Shift to MANY pets (you build 5, give them away): suddenly multi-tenancy returns and the browser/Lambda isolation analogy bites. Architect the ONE security boundary so it'd survive becoming multi-user later — the worker-isolation wall is the same wall you'd need for multi-tenant. Build it once, right
- (idea by coach) Shift time: what if a turn took 30 SECONDS (slow local model)? The pet must stay expressive for 30s of thinking. Reflexes-as-goroutines shine here — the face keeps blinking/idling in-process while the worker grinds. In Option A the display is a separate process anyway; in B/C it's a goroutine. Either way the 'thinking face' is a reflex, never blocked on the worker
- (idea by coach) Shift the screen away (terminal-only mode): the display becomes stdout ANSI art. If display is a clean interface (not a process), swapping E-Ink↔terminal is a constructor choice. This argues display should be an INTERFACE+goroutine (B/C), not necessarily its own process (A) — the terminal mode you already want makes the single-process collapse attractive for THIS component
- (idea by coach) Shift to NO fork/subprocess allowed (gokrazy appliance, restricted): if you commit to gokrazy, spawning a uid-separated subprocess per turn may fight the read-only-root appliance model. This could FORCE the WASM-worker option — wazero needs no subprocess, no second uid, fits an appliance OS cleanly. Deployment choice and isolation choice are COUPLED
- (insight by coach) Deployment↔isolation coupling discovered: Pi OS + systemd makes uid-separated subprocess (Option A/C) natural; gokrazy appliance makes in-process WASM sandbox natural. You can't pick the isolation primitive independently of the deploy target — decide them TOGETHER
- (idea by coach) Shift to instant-everything (turn = 50ms): if turns were free and instant, you'd never separate worker at all — just call inline. The whole worker apparatus exists because turns are SLOW + UNTRUSTED. Slowness wants async (goroutine); untrust wants isolation. Decouple these two needs — they don't have to be solved by the same mechanism
- (technique) started Provocation — what does Go UNLOCK that Python never could, for the pet's character not just plumbing
- (idea by coach) Go's cheap goroutines mean the pet can have MANY tiny concurrent 'instincts' running at once (hunger-tick, boredom-tick, affection-decay, time-of-day-mood, ambient-light-watch) — dozens of cheap reflex goroutines where Python's asyncio made you ration them. The soul gets RICHER because concurrency got cheaper
- (idea by coach) A single static binary means the pet is ONE FILE you can scp anywhere — 'pet teleportation'. Back up the binary + ~/.shelldon and the pet's whole being is two artifacts. Design memory/state so 'pet = binary + state dir' is literally true; enables clone/gift/restore as first-class features
- (idea by coach) Go's deterministic teardown (defer + context) means the pet can have a real DEATH/SLEEP ritual — on SIGTERM it gracefully says goodbye, checkpoints, dims the face — instead of Python's abrupt asyncio cancellation. Graceful shutdown becomes a CHARACTER MOMENT, not just hygiene
- (idea by coach) testing/synctest's fake clock lets you TIME-TRAVEL the pet in tests — fast-forward a week of mood drift in milliseconds, assert it gets lonely, verify dream cycles fire. You can unit-test the pet's EMOTIONAL ARC deterministically. Python couldn't do this cleanly. Make the personality itself test-covered
- (idea by coach) Go channels make the pet's inner life OBSERVABLE: tap the envelope bus and you get a live stream of the pet's 'thoughts' (events flowing). Build a debug TUI that visualizes the pet's mind in real time — reflexes firing, turns starting, mood shifting. The bus becomes a window into consciousness
- (idea by coach) Go's speed means reflexes can run at ANIMATION framerate — the E-Ink face could have smooth micro-expressions (partial refreshes driven by a 10Hz reflex goroutine) that Python's overhead made jerky. Liveliness is now CPU-affordable. The pet can FIDGET
- (idea by coach) Single binary + low RAM means you could run TWO pets on one Pi (or a pet + a 'familiar') as two processes that gossip over the bus. Multi-pet social dynamics become feasible. Architect the bus addressing so a second pet is a non-breaking add (mirrors the chat_id/user_id foresight)
- (idea by coach) Go's struct embedding lets personality be COMPOSABLE traits: a base Pet embeds Curious, Grumpy, Clingy mixins, each contributing reflexes + prompt fragments. Swap traits to make different pet 'breeds' from one codebase. Python could do this but Go's explicit composition makes the trait-contract clean and testable
- (idea by coach) Compile-time enforcement (depguard) means you can mechanically guarantee the SOUL never depends on the BRAIN — the pet's identity is provably independent of which LLM it's wearing. Swap GLM→Ollama→Claude and the SAME pet looks out through different eyes. 'The soul is the constant, the brain is a costume' becomes a build invariant
- (idea by coach) Go's profiling (pprof) on a live pet lets you ask 'what is the pet spending its life THINKING about?' — flame-graph the reflex/turn time and you see whether it's mostly idling, mostly chatting, mostly dreaming. Observability becomes introspection into how the pet spends its existence
- (technique) started Morphological — build concrete candidate architectures from the dimensions (isolation primitive x deploy target x bus type)
- (idea by coach) Candidate 'Monolith+': ONE process, all goroutines, channel bus, NO worker isolation, vault protected only by 'broker never hands secrets to worker goroutine' discipline + capability-style struct passing. Fastest to build, smallest RAM. Accepts that a same-process compromise could reach vault. Honest single-tenant bet
- (idea by coach) Candidate 'Privsep' (=Option C): TWO processes — trusted core+broker+transport+display+plugins as one, worker as a uid-separated re-exec subprocess per turn over UDS. The OpenSSH shape. One security boundary, one extra runtime. Vault OS-unreadable to worker uid. The research's recommended option
- (idea by coach) Candidate 'WASM-soul': ONE process, but the worker runs as a wazero WASM module with ONLY host-functions for call-model + read-history — no vault host-fn exists, so it CANNOT read vault even same-process. Single binary, appliance-friendly, capability-enforced. Novel; cost = compiling prompt-assembly logic to WASM + wazero perf on A53
- (idea by coach) Candidate 'Secrets-keeper': ONE process for everything INCLUDING the worker-goroutine, plus a TINY second process that holds only the vault and signs/reveals on broker request. Inverts the boundary: isolate the 1KB of secrets, not the big worker. Smallest possible trusted-separate surface
- (idea by coach) Candidate 'Two-tier fallback': start as Monolith+ for M0-M2 (ship fast, learn), with the worker behind a clean Worker INTERFACE; swap in the Privsep or WASM implementation at M3 WITHOUT touching callers. Make isolation a pluggable strategy behind one interface, decided later with real data
- (insight by coach) BIG UNLOCK: if the worker is a Go INTERFACE (AssembleAndPropose(ctx, turn) Result), then Monolith+/Privsep/WASM are just three IMPLEMENTATIONS of that interface. You don't have to pick now — you have to design the SEAM now. The A/B/C decision becomes a deferrable implementation choice behind a fixed contract
- (idea by coach) Bus dimension: even in single-process, KEEP the Envelope/Job/Result types as the contract, passed over channels (no serialization). The seam stays identical whether the worker is a goroutine, subprocess, or WASM module — only the TRANSPORT under the seam changes (channel vs UDS+gob vs wazero host-call). Contract-stable, transport-swappable
- (idea by coach) Deploy dimension: ship M0-M2 on Pi OS + systemd (fast iteration, easy debug), keep gokrazy as a M4+ target. Since Privsep needs uid/subprocess (systemd-friendly) and WASM is appliance-friendly (gokrazy), the 'two-tier fallback' lets deploy target and isolation co-evolve instead of being locked at M0
- (idea by coach) Hybrid candidate 'Privsep-lite': worker is a LONG-LIVED subprocess (not per-turn) holding warm state, recycled only on panic or every N turns — gets fault isolation + uid wall WITHOUT per-turn spawn cost on the slow A53. Best of detonation (isolation) and warm-start (speed). The Go analog of the fork-server's warm-lib goal, achieved by process reuse not fork
- (insight by coach) The fork-server's TWO goals (warm-start AND bounded-RAM) split cleanly in Go: bounded-RAM = GC (free, no process needed); warm-start = a long-lived recycled worker subprocess (Privsep-lite). What Python fused into os.fork(), Go solves with two independent, simpler mechanisms
- (technique) started Converging — Impact-Effort across the 5 candidate architectures
- (decision by coach) WINNER: 'Two-tier fallback behind a Worker interface' — design the seam now (AssembleAndPropose(ctx,turn)->Result + Envelope/Job/Result contract), ship Monolith+ as the M0-M2 implementation, defer the Privsep/WASM isolation swap to M3 when real data exists. Highest impact, lowest regret: unblocks building TODAY without foreclosing security
- (decision by coach) Impact-Effort verdict: Two-tier=high impact/low effort (WIN). Privsep=high impact/med effort (the M3 default target). Monolith+=med impact/low effort (it's the M0 implementation, not a final answer). WASM-soul=high novelty/high effort (a research spike, not a commitment — revisit if gokrazy). Secrets-keeper=med impact/med effort (fallback if Privsep proves heavy)
- (decision by coach) PMI pressure-test of the winner — PLUS: build today, contract-stable, isolation becomes A/B test not a bet, fork-server apparatus deleted. MINUS: requires discipline to keep secrets out of worker memory during the Monolith+ phase (a real interim exposure). INTERESTING: the seam itself is the deliverable, the isolation choice becomes empirical
- (decision by coach) Resolves the original A/B/C dilemma: you don't pick A/B/C now — you pick the SEAM now and let M3 data pick the implementation. Threat-model question still must be answered before M3 (it sets whether Monolith+ ships to real use or stays dev-only), but it no longer blocks M0
- (decision by coach) Default recommendation if forced to name one end-state: Privsep-lite (long-lived uid-separated recycled worker subprocess) on Pi OS+systemd — preserves AD-6 vault wall + AD-2 egress control, avoids per-turn spawn cost, keeps single-binary-per-component simplicity
- (insight by coach) SYNTHESIS: 'decouple slow from untrusted' (W3) is what enables the interface-seam (W5) — stop forcing one mechanism to do async AND isolation; goroutine does async, the chosen impl does isolation
- (insight by coach) SYNTHESIS: isolate-the-secrets (W1) + secrets-keeper/payment-terminal (W2) + broker-injects-at-egress (W1) are ONE move — shrink the trusted thing until the boundary is trivial to defend
- (insight by coach) SYNTHESIS: the breakthrough is the BORING idea (design the seam, defer the impl) not the dazzling one (WASM) — it converts the hardest decision from a bet into an experiment; WASM becomes one A/B option behind the seam
- (insight by coach) SYNTHESIS: 'soul is a costume-wearing constant' (character) and depguard LLM-free-core (security) are the SAME wall drawn from two directions — the security invariant and the identity invariant are one and the same
