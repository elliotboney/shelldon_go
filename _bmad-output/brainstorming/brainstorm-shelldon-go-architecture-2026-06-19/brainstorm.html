THE LIVE WIRE

How do you keep the untrusted brain away from the secret heart?

The LLM worker assembles prompts from web-influenced text. If injected text talks it into leaking the vault, the pet betrays itself. The whole session orbits one boundary question.

01 First Principles Strip the v2 spine to invariants. Challenge every "must" from Go's vantage.

The v2 spine stood on three legs. Go kicks out two.

BOUNDED RAM
GC gives it free
WARM REFLEXES
just ticker goroutines
SECURITY WALL
★ only leg left
coachThe fork-server was v2's load-bearing bet to protect bounded RAM — Go gives that free, so the bet's value is now zero. The question it answered re-emerges as a fault question, not a memory one.
coachReframe "≤1 worker": in Go RAM isn't the constraint, so keep ≤1 not for megabytes but for coherence (one train of thought) + cost (budget). The pet has one mind, not few bytes.
coachThird isolation primitive: the worker as a WASM sandbox via wazero — prompt-assembly runs in linear memory with no ambient authority. Sits between goroutine (no isolation) and subprocess (OS isolation), stays single-binary.
coachInversion: don't isolate the worker — isolate the secrets. A tiny secrets-keeper holds the vault; broker queries it. Blast radius of a collapse becomes "everything except secrets" — a sharper boundary.
coachSecrets-never-in-worker: the broker injects creds at the egress moment; the worker never receives them. Single-process makes this necessary-but-insufficient — the cleanest argument for keeping exactly ONE process boundary.
v2 optimized for Python's failure modes (OOM, GIL, cheap fork). Go should optimize for Go's: goroutine leaks, shared-memory races, panic propagation. Different enemies → different shape.
The vault requirement, said precisely: not "separate process" but "the thing touching web-influenced text cannot read secrets." A goroutine shares the address space, so it's insufficient by definition. The real requirement is address-space separation — only the kind is open (process vs WASM linear memory).
02 Analogical Thinking Who else runs untrusted compute behind a trust boundary — and what did they pick?
Web Browser → C/A
A tab = separate OS process; the kernel holds cookies. Chrome chose process isolation because shared memory wasn't safe. Worker = a tab.
AWS Lambda weak
Untrusted code in Firecracker microVMs, one per invocation. But that's multi-tenancy; shelldon is single-tenant.
Game Mods → WASM
Mods run in a sandboxed Lua VM inside the host — capability-restricted, not a process. The wazero pedigree.
Payment Terminal → keeper
A secure element holds keys on separate silicon; the app never sees the key, only asks it to sign. Broker-as-keeper, in hardware.
qmail / postfix → A
A monolith split into tiny single-uid processes over pipes. Bernstein privilege separation — 25-year-proven. The spine is qmail-shaped.
OpenSSH privsep ★ steal
Network-facing code in a chroot+unpriv child; a monitor holds privilege & validates requests. Worker proposes, monitor disposes.
AV Detonation overkill
Suspicious files detonated in a throwaway VM. "Die after a turn" is detonation — but our Result is trusted-after-validation. Buys fault-reset, not security.
seL4 / Fuchsia → WASM
No ambient authority — only what you were handed a capability for. Hand the worker call-model + read-history, nothing else. wazero host-fns enforce it.
The analogies split by tenancy ⟶ multi-tenant (browser, Lambda) spends heavily on process/VM isolation; single-tenant embedded (game mods, your desk pet) accepts in-process sandboxing. shelldon is single-tenant — the WASM sandbox is more defensible than the browser analogy first suggests.
03 Constraint Shifting Warp the givens — RAM, network, hardware, time — and watch the architecture fall out.
RAM 512MB → 4GB
Frugality apparatus is pointless — run one fat process. Proves frugality is hardware-contingent: make it a build flag, not a bone.
RAM 512MB → 256MB
Go's ~5-10MB runtime floor per process bites. 5 processes = 50MB pure overhead. Tight RAM argues for single-process collapse.
NET online → zero
Degrade to Ollama-LAN or reflex-only. "No brain available" must be first-class, not an error. Offline is a personality mode.
FLEET 1 pet → 5 pets
Multi-tenancy returns; the browser analogy bites. Build the one wall so it survives going multi-user. Build it once, right.
TURN fast → 30s
The pet must stay expressive while it grinds. Reflexes-as-goroutines keep the face blinking, never blocked on the worker.
SCREEN E-Ink → terminal
Display becomes stdout ANSI art. Make display an interface+goroutine, swappable by constructor — favours single-process.
DEPLOY systemd → gokrazy
No fork/subprocess on a read-only appliance could force WASM — wazero needs no second uid. Deploy & isolation are coupled.
TURN slow → 50ms
If instant & free, never separate at all. Slowness wants async; untrust wants isolation — decouple the two needs.
Key tension: more processes = more isolation BUT more runtime overhead (each pays the floor). On 512MB, Option C (2 processes) is the sweet spot — one security boundary at the cost of one extra runtime floor.
Deployment ↔ isolation coupling: Pi OS + systemd makes uid-separated subprocess natural; gokrazy appliance makes in-process WASM natural. You can't pick the isolation primitive independently of the deploy target — decide them together.

The fork-server's two goals split cleanly in Go: bounded-RAM = GC (free, no process); warm-start = a recycled worker subprocess (Privsep-lite). What Python fused into os.fork(), Go solves with two simpler mechanisms.

SYNTHESISwhere the ideas were secretly solving each other

1
Decouple slow from untrusted

W3 → W5. Stop forcing one mechanism to do async AND isolation. The goroutine does async; the chosen impl does isolation. That separation is what enables the interface-seam.

2
Shrink the trusted thing

W1 + W2 + W1. Isolate-the-secrets, the secrets-keeper / payment-terminal, and broker-injects-at-egress are one move — shrink the trusted thing until the boundary is trivial to defend.

3
The boring idea won

The breakthrough is the dull one — design the seam, defer the impl — not the dazzling WASM. It converts the hardest decision from a bet into an experiment. WASM becomes one option behind the seam.

4
One wall, two directions

"Soul is a costume-wearing constant" (character) and depguard's LLM-free core (security) are the same wall drawn from two sides. The identity invariant and the security invariant are one.
